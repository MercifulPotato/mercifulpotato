Item(by='a1369209993', descendants=None, kids=None, score=None, time=1609707484, title=None, item_type='comment', url=None, parent=25623149, text='&gt; Most of those will still look like corrupted data,<p>Not if you&#x27;re using a typical 72-bit SECDED code[0].<p>You have two error indicators: a summary parity bit (even number of errors: 0,2,etc vs odd number of errors: 1,etc), and a error index: 0 for no errors, or the bitwise xor of the locations each bit error.<p>For a triple error at bits a,b, and c, you&#x27;ll have summary parity of 1 (odd number of errors, assumed to be 1), and a error index of a^b^c, in the range 0..127, of which 0..71[1] (56.25%, a clear albeit not overwhelming majority) will correspond to legitimate single-bit errors.<p>0: <a href="https:&#x2F;&#x2F;en.wikipedia.org&#x2F;wiki&#x2F;Hamming_code#Hamming_codes_with_additional_parity_(SECDED)" rel="nofollow">https:&#x2F;&#x2F;en.wikipedia.org&#x2F;wiki&#x2F;Hamming_code#Hamming_codes_wit...</a><p>1: or 72 out of 128 anyway; the active bits might not all be assigned contiguous indexes starting from zero, but it doesn&#x27;t change the probability and it&#x27;s simpler to analyse if summary is bit 0 and index bit i is substrate bit 2^i.')
