It looks kinda interesting.<p>Actual title: &quot;Dial String Parameter for the\n        Session Initiation Protocol Uniform Resource Identifier&quot;<p>Explainer from link:<p>&quot;\nStatus of This Memo<p><pre><code>   This document specifies an Internet standards track protocol for the\n   Internet community, and requests discussion and suggestions for\n   improvements.  Please refer to the current edition of the &quot;Internet\n   Official Protocol Standards&quot; (STD 1) for the standardization state\n   and status of this protocol.  Distribution of this memo is unlimited.\n</code></pre>\nCopyright Notice<p><pre><code>   Copyright (C) The IETF Trust (2007).\n</code></pre>\nAbstract<p><pre><code>   RFC 3966 explicitly states that &#x27;tel&#x27; URIs may not represent a dial\n   string.  That leaves no way specify a dial string in a standardized\n   way.  Great confusion exists with the SIP URI parameter &quot;user=phone&quot;,\n   and specifically, if it can represent a dial string.  This memo\n   creates a new value for the user parameter &quot;dialstring&quot;, so that one\n   may specify &quot;user=dialstring&quot; to encode a dial string as a &#x27;sip:&#x27; or\n   &#x27;sips:&#x27; URI.</code></pre>')
