Item(by='dilap', descendants=None, kids=[25644398], score=None, time=1609803259, title=None, item_type='comment', url=None, parent=25632033, text='&gt; Just thinking out loud, given Zig&#x27;s errors primarily allow the compiler to enforce that errors are handled, and exhaustively, beyond that why can&#x27;t error payloads just be passed as an input argument pointer to the function call?<p>Yeah, that&#x27;s definitely a reasonable way to do it, and actually Andrew gives an example of this:<p><a href="https:&#x2F;&#x2F;github.com&#x2F;ziglang&#x2F;zig&#x2F;issues&#x2F;2647#issuecomment-589829306" rel="nofollow">https:&#x2F;&#x2F;github.com&#x2F;ziglang&#x2F;zig&#x2F;issues&#x2F;2647#issuecomment-5898...</a><p>(Andrew&#x27;s example is <i>slightly</i> different in that rather than an explicit argument, it&#x27;s an optional field in one of the arguments.)<p>So it&#x27;s definitely not like crazy bad or anything, but also definitely less ergonomic than if you could directly return a payload with the error, and this is enough friction that it feels like you&#x27;ll end up not having error information when you want it (both as an API consumer and eventually as an end-user looking at an error message).<p>But, I&#x27;m still very new to Zig, so perhaps my instincts on that are wrong. :-)')
