Item(by='j-pb', descendants=None, kids=[25872402, 25872009], score=None, time=1611326390, title=None, item_type='comment', url=None, parent=25870772, text='ULID has a serious security flaw though.\nIn order to try to improve performance &#x2F; increase prefix similarity, there is a mode where it reuses the random value, and just increments it, for each new ULID.<p>With purely random UUID, you can be pretty sure that nobody can guess them, so they&#x27;re essentially identifier and access token in one.<p>However once you can easily predict the next uuid from the previous one (as by incrementing the random part by one), the access token property vanishes.<p>At least this proposal doesn&#x27;t make the same mistake,\nhowever 80 bit&#x27;s isn&#x27;t THAT much entropy.\n(Remember every bit counts with exponential growth,\nand 48 bits less, means 281474976710656 times easier to guess.)<p>So it boils down to. Would you be ok to secure all your data with 10 character one time passwords, which might not be guessed all at once, but where guessing SOME is sufficient to get owned?<p>Death by a thousand paper-cuts.')
