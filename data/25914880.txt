Item(by='gbrown_', descendants=None, kids=[25915200, 25915212, 25915813], score=None, time=1611662955, title=None, item_type='comment', url=None, parent=25913558, text='I&#x27;m really not a fan of the decision to overhaul Vim script. It seems this is driven from a desire for Vim script to be faster [1]. I&#x27;ve not needed Vim script to be &quot;faster&quot; but I don&#x27;t use very complex plugins. That said a lot of Vim script is written in sub-optimally. Using a more functional approach is faster (though still not &quot;fast&quot;), e.g.<p><pre><code>    $ cat loop.vim\n    let sum = 0\n    for i in range(1, 2999999)\n      let sum += i\n    endfor\n    quit\n    \n    $ cat fold.vim\n    call eval(join(range(1, 2999999), &#x27;+&#x27;))\n    quit\n    \n    $ time vim -u NONE -S loop.vim\n    vim -u NONE -S loop.vim  7.08s user 0.09s system 99% cpu 7.198 total\n    \n    $ time vim -u NONE -S fold.vim\n    vim -u NONE -S fold.vim  2.03s user 0.13s system 99% cpu 2.166 total\n</code></pre>\nI think its cryptic reputation comes form the use of normal mode commands and regex within scripts. There certainly are warts, like the user&#x27;s &#x27;ignorecase&#x27; setting permeating to many places in the runtime. Which I&#x27;d love to see fixed and whilst may also represent a breaking change for some scripts would be far more easily adoptable.<p>[1] <a href="https:&#x2F;&#x2F;github.com&#x2F;brammool&#x2F;vim9&#x2F;blob&#x2F;master&#x2F;README.md" rel="nofollow">https:&#x2F;&#x2F;github.com&#x2F;brammool&#x2F;vim9&#x2F;blob&#x2F;master&#x2F;README.md</a>')
