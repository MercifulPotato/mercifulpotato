Item(by='partialstate', descendants=None, kids=None, score=None, time=1607716834, title=None, item_type='comment', url=None, parent=25377620, text='The goal of CEL is fast, scalable, and portable expression evaluation.<p>Fast - CEL runs without the need for sandboxing, making it much faster than sandboxed solutions like WebAssembly, Lua, and embedded JavaScript.<p>Scalable - Features like variables and functions would make CEL more expressive, but also less scalable as it&#x27;s easy to write a few lines of code with functions that consume exponential amounts of memory and compute. CEL is simply the expression and nothing more.<p>Portable - CEL is implemented in Go[0], C++[1], and Python[2] with Java open sourcing in development. There is a public codelab[3] available for Go if anyone is interested. There is also a conformance suite in CEL-Spec to ensure consistent behavior between runtimes and environments. Our objective is to make it possible to bring CEL to K8s, J2EE apps, and C++ proxies. Evaluate at line-rate everywhere. Personally, I hope someone tries to make CEL work on IoT devices some day too.<p>Where? - CEL is usually embedded into larger projects rather than being the one stop shop for solving a particular kind of problem. For example, CEL Policy Templates[4] has an opinionated way of using CEL to validate&#x2F;evaluate YAML configs. Most of the time CEL is part of a service API.<p>In addition to being used in Firebase&#x27;s Cloud Firestore &#x2F; Cloud Storage security rules, it is also used in several other Google Cloud services: \n- Cloud Armor[5]\n- IAM Conditions[6]\n- Cloud Healthcare Consents[7] \n- Cloud Build Notifiers[8]\n- Security Token Service[9]\n- Access Levels[10], and more.<p>CEL is also used in some
