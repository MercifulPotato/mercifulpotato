Yes, upping the site with articles and examples is on the TODO :)<p>In the SSH module doc [0] there are a few examples, such as:<p><pre><code>  space -m ssh &#x2F;ssh&#x2F; -e SSHHOST=jump1.example.com,jump2.example.com,jump3.example.com,destination.example.com -e SSHCOMMAND=&quot;ls -l&quot;\n</code></pre>\nOne very fun feature is that you can tab auto-complete filenames and container names and also files inside container on a remote host:<p><pre><code>  space -m ssh -e SSHHOST=1.2.3.4 -e SSHPORT=2233 -e SSHUSER=rabbit &#x2F;wrap&#x2F; -m docker &#x2F;wrap_exec&#x2F; -e DOCKERCONTAINER=[tab tab]\n</code></pre>\nThen:<p><pre><code>  space -m ssh -e SSHHOST=1.2.3.4 -e SSHPORT=2233 -e SSHUSER=rabbit &#x2F;wrap&#x2F; -m docker &#x2F;wrap_exec&#x2F; -e DOCKERCONTAINER=something -m file &#x2F;cat&#x2F; -e file=&#x2F;home&#x2F;[tab tab]\n</code></pre>\nNote that server jumping (over firewalls) is perfectly viable in the last example too, by adding more hosts (comma separated). Note that it can be important to balance host names to user names (see the doc [0]).<p>In the examples space <i>wraps</i> module functionality where the last module <i>file</i> is the one finally running inside the container, wrapped first by the `ssh` module then the `docker` module.<p>It is totally agent-less so nothing is uploaded to the server, it&#x27;s just like SSHing in and running commands.<p>If you add the `-d` flag to space it will dump the script to output, which you can save to a .sh file and run later (without needing space.sh at all).\nSpace it self requires bash, but the output runs with POSIX shells, so it also works with dash&#x2F;ash&#x2F;busybox, etc which is good because servers&#x2F;containers often doesn&#x27;t come with bash, but simpler ash&#x2F;dash instead.<p>Thanks for showing interest, this makes me keen on pushing a few fixes and updates I have pending.
