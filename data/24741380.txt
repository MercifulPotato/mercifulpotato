Item(by='txutxu', descendants=None, kids=None, score=None, time=1602359377, title=None, item_type='comment', url=None, parent=24738276, text='&gt; - #!&#x2F;usr&#x2F;bin&#x2F;env &gt; #!&#x2F;bin&#x2F;bash for portability<p>Yes... but do not take that as a &quot;cargo cult script shebang&quot;.<p>If you&#x27;re a sysadmin writing a script for a company with 2k linux servers, that has a policy of &quot;we only use linux version Foo X&quot;... and we do not use other bash in the system than &#x2F;bin&#x2F;bash (no bash compiled by hand, no multiple versions of bash, etc)... then portability via &quot;env&quot; does not make sense.<p>If you have two laptops and a raspberry at home, with debian or arch, and you write a script for yourself... then portability via &quot;env&quot; does not make sense.<p>And last but not least... using env is slower.<p>See:<p><pre><code>    strace -fc &#x2F;bin&#x2F;bash -c &#x27;:&#x27;\n</code></pre>\nVs<p><pre><code>    strace -fc &#x2F;usr&#x2F;bin&#x2F;env bash -c &#x27;:&#x27;\n</code></pre>\nOn my system, that&#x27;s 92 syscalls and 3 errors, Vs 152 syscalls and 8 errors.<p>Just to start procesing.<p>Diferent levels of system bloat (environment, library paths, etc) can give different results than my example.<p>And as others said... if you&#x27;re not using GNU&#x2F;bash syntax and the script is really simple, the best for portbility is to go with &#x2F;bin&#x2F;sh.<p><pre><code>    strace -fc &#x2F;usr&#x2F;bin&#x2F;env sh -c &#x27;:&#x27;\n</code></pre>\nOn my system 41 syscalls and 1 error... (and less RAM, CPU and pagefaults).<p>If you&#x27;re not using associative arrays, array indexes, non POSIX builtin options, and other bash extensions... if the script is just to join a few commands and variables... it pays the effort to write it in simple sh, both, for portability and performance.')
